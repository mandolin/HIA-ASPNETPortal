# 发布回滚指南

本文用于 P13.1 起的文件系统发布包回滚准备。它是发布检查清单的配套材料，不替代真实 IIS、数据库、备份平台或企业变更系统的操作规程。

## 基本原则

1. 每次发布前必须能定位“上一版发布包、当前发布包、外置配置备份、数据库迁移记录和验证证据”。
2. 回滚操作不得依赖口头记忆；发布包、配置和数据库的回滚边界必须提前写清。
3. 真实密码、连接串、Token、Cookie、证书私钥和生产配置不得进入 Git、WorkZone 或发布证据包。
4. 若数据库迁移不可自动回滚，应在发布前明确“恢复备份”或“人工数据修复”的责任人和窗口。

## 文件回滚

- [ ] 保留上一版 FileSystem 发布包或可重新生成上一版发布包的提交号。
- [ ] 记录当前发布包路径、发布包 manifest 和 Git 提交号。
- [ ] 发布时不要把 `work-zone/`、`temp/`、上传业务文件、AI 日志、真实配置或生成证据放入站点目录；`Uploads/web.config` 可作为上传目录安全配置随包发布。
- [ ] 回滚前备份当前站点目录，避免覆盖后丢失现场证据。
- [ ] 回滚后运行首页、登录、后台健康页和核心业务页 smoke。

## 配置回滚

- [ ] 记录 Web.config transform 来源和目标 profile，例如 Dev/Test/Prod/Scan/LegacyIe。
- [ ] 保留外置配置目录备份，但不要把真实配置内容写入证据包。
- [ ] 记录 `ExternalCfgPath`、环境变量键名、机器级配置位置和证书引用方式。
- [ ] Cookie SSL、HSTS、customErrors、安全 header、machineKey 变更需要可单独回滚。
- [ ] 配置回滚后检查通用错误页、登录、日志和系统健康页。

## 数据库回滚

- [ ] 发布前确认数据库完整备份可用。
- [ ] 记录已执行迁移脚本、manifest、执行顺序、执行人和执行时间。
- [ ] 若迁移脚本包含回滚脚本，先在隔离测试库验证回滚路径。
- [ ] 若迁移不可逆，发布前必须确认恢复备份窗口和业务停机影响。
- [ ] 数据库回滚后运行只读兼容检查和关键业务 smoke，不直接在生产窗口尝试未验证脚本。

## 证据保留

- [ ] 保留发布前门禁输出：发布资源、公开文档、默认凭据、合规基线和必要业务验收。
- [ ] 保留发布包 manifest：文件清单、SHA256、排除项检查和版本信息。
- [ ] 保留发布后 smoke：访问入口、登录、后台权限、日志审计和错误页事件编号。
- [ ] 若发生回滚，记录触发原因、事件编号、回滚开始/结束时间和未解决风险。

## 后续补证

真实 IIS、TLS、应用池 ACL、目标 SQL Server 2016/2017/2019、企业扫描工具和业务负责人签收，都属于环境触发型证据。没有目标环境时不得宣称已经完成，只能记录为待补证项。
