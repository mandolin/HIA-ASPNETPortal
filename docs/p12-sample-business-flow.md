# P12 员工资料更正样板业务路径

本文说明 P12 当前用于演示和回归的最小企业业务路径：普通员工通过已有账号登录，基于有效员工绑定提交资料更正请求；管理员在后台审核请求，并通过待办、审计和员工目录进行追踪。

## 边界

1. 本路径用于开发、测试和演示，不是生产初始化流程。
2. 样例数据不创建用户密码，不保存任何凭据。
3. 员工号只是登录标识之一，不是独立密码或独立认证凭据。
4. 当前只覆盖字段级文本更正，不处理附件、手机号、身份证号、薪资、绩效等高敏信息。
5. 当前待办是轻量能力，不包含转办、委托、会签、通知或通用流程设计器。

## 自动侧验收

运行 P12.5 证据包脚本：

```powershell
dev/scripts/New-PortalP12EvidencePackage.ps1
```

脚本会编排以下只读检查，并默认执行一次解决方案构建：

1. P12.2 业务身份门禁。
2. P12.3 轻量待办门禁。
3. P12.4 业务权限与审计门禁。
4. 解决方案构建。

证据默认写入 `work-zone/dev/evidence/p12.5/`。如果没有 `work-zone/`，会写入 `temp/evidence/p12.5/`。

## 样例数据 SQL

生成开发/测试用样例 SQL：

```powershell
dev/scripts/New-PortalP12SampleScenarioSql.ps1 -PortalUserName P12EmployeeDemo
```

默认输出到：

```text
temp/p12.5/PortalP12SampleScenario.sql
```

执行前需要先在开发库或测试库中准备一个名为 `P12EmployeeDemo` 的门户测试用户，并确保业务迁移脚本已经应用。生成的 SQL 会补齐组织、员工、用户资料和账号员工绑定，但不会创建用户或修改密码。

## 手工回归路径

1. 用普通员工测试账号登录。
2. 打开员工资料确认模块，确认能看到员工号、姓名、组织等资料。
3. 打开员工资料更正请求模块，选择一个低敏字段，填写建议值并提交。
4. 用管理员账号登录后台，打开 `Admin/EmployeeProfileCorrectionRequests.aspx`。
5. 确认刚才提交的请求出现在列表中。
6. 打开 `Admin/WorkItems.aspx`，确认该请求对应的轻量待办存在。
7. 在资料更正后台执行批准、驳回或关闭中的一个动作。
8. 打开运营审计页，确认提交和审核动作能被追踪。
9. 再次打开待办后台，确认对应待办状态已经同步为完成或取消。

## 预期结果

1. 员工工号可作为登录标识，但仍使用已有测试用户密码。
2. 未绑定员工的账号不能提交资料更正请求。
3. 资料更正提交成功后会产生运营审计记录和轻量待办。
4. 管理员审核成功后会产生审核审计记录，并同步待办状态。
5. 当前权限仍保留 `Admins` 兼容，但代码入口已经使用 P12.4 细粒度业务权限。

## 后续深化方向

1. 建立角色权限 UI 后，将当前细粒度权限分配给 HR、业务审核员和只读观察员等角色。
2. 将读取类审计、审计详细级别和审计保留策略纳入系统设置与发布时配置。
3. 在后续 Workflow 周期中补转办、委托、会签、通知和更多业务对象。
